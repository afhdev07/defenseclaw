# DefenseClaw — Observability Guide

Sending traces and metrics from DefenseClaw to Splunk Observability Cloud via OTLP HTTP.

## Prerequisites

- Gateway binary built: `make gateway`
- A Splunk Observability Cloud access token
- Your Splunk realm (e.g., `us1`, `us0`)

## Quick Start

### Option 1: Using start.sh (recommended)

```bash
./scripts/start.sh --enable-telemetry --access-key <SPLUNK_TOKEN> --app-name pavan-defenceapp
```

With a specific realm:

```bash
./scripts/start.sh --enable-telemetry --access-key <SPLUNK_TOKEN> --realm us0 --app-name pavan-defenceapp
```

Via Make:

```bash
make start ARGS="--enable-telemetry --access-key <SPLUNK_TOKEN> --app-name pavan-defenceapp"
```

### Option 2: Direct invocation

```bash
DEFENSECLAW_OTEL_ENABLED=true \
SPLUNK_ACCESS_TOKEN="<SPLUNK_TOKEN>" \
DEFENSECLAW_OTEL_TRACES_ENDPOINT="ingest.us1.observability.splunkcloud.com" \
DEFENSECLAW_OTEL_TRACES_PROTOCOL="http" \
DEFENSECLAW_OTEL_TRACES_URL_PATH="/v2/trace/otlp" \
DEFENSECLAW_OTEL_METRICS_ENDPOINT="ingest.us1.observability.splunkcloud.com" \
DEFENSECLAW_OTEL_METRICS_PROTOCOL="http" \
DEFENSECLAW_OTEL_METRICS_URL_PATH="/v2/datapoint/otlp" \
OTEL_SERVICE_NAME="pavan-defenceapp" \
./defenseclaw-gateway
```

For a different realm, replace `us1` with your realm (e.g., `us0`, `eu0`).

### Gateway only (no telemetry)

```bash
./scripts/start.sh
# or
./defenseclaw-gateway
```

## Architecture

```
                        Without telemetry
                        ┌──────────────┐
                        │  DefenseClaw │
                        │   Gateway    │  (OTel disabled, no export)
                        └──────────────┘

                        With telemetry
┌──────────────┐  HTTP traces   ┌──────────────────────┐
│  DefenseClaw ├───────────────►│ Splunk APM            │
│   Gateway    │                │ /v2/trace/otlp        │
│              │  HTTP metrics  ├──────────────────────┤
│              ├───────────────►│ Splunk IM             │
│              │                │ /v2/datapoint/otlp    │
│              │  HTTP logs     ├──────────────────────┤
│              ├ ─ ─ ─ ─ ─ ─ ─►│ Splunk Log Observer   │
│              │  (optional)    │ /v1/log/otlp          │
└──────┬───────┘                └──────────────────────┘
       │
       │  Audit events (HEC)    ┌──────────────────────┐
       └───────────────────────►│ Splunk Enterprise     │
                                │ /services/collector   │
                                └──────────────────────┘
```

No sidecar collector needed — the gateway exports directly to Splunk via OTLP HTTP.

## Signal Routing

| Signal | Protocol | Endpoint |
|--------|----------|----------|
| Traces | HTTP | `ingest.<realm>.observability.splunkcloud.com/v2/trace/otlp` |
| Metrics | HTTP | `ingest.<realm>.observability.splunkcloud.com/v2/datapoint/otlp` |
| Logs (OTel) | HTTP | `ingest.<realm>.observability.splunkcloud.com/v1/log/otlp` (manual setup) |

| Signal | Destination |
|--------|-------------|
| Traces | Splunk APM |
| Metrics | Splunk Infrastructure Monitoring |
| Logs (OTel) | Splunk Log Observer (if configured) |
| Audit Events (HEC) | Splunk Enterprise / Splunk Cloud (separate pipeline) |

## Logs & Events

DefenseClaw has two independent pipelines for log/event data. They are **not connected** to each other.

### OTel Logs (otel.logs)

Structured OTel log records emitted by the gateway: lifecycle events, scan results, runtime alerts. These use the standard OTLP log exporter.

`start.sh` does **not** configure a logs endpoint — only traces and metrics are routed by default. To enable OTel logs to Splunk Log Observer, set the logs env vars manually:

```bash
DEFENSECLAW_OTEL_ENABLED=true \
SPLUNK_ACCESS_TOKEN="<SPLUNK_TOKEN>" \
DEFENSECLAW_OTEL_TRACES_ENDPOINT="ingest.us1.observability.splunkcloud.com" \
DEFENSECLAW_OTEL_TRACES_PROTOCOL="http" \
DEFENSECLAW_OTEL_TRACES_URL_PATH="/v2/trace/otlp" \
DEFENSECLAW_OTEL_METRICS_ENDPOINT="ingest.us1.observability.splunkcloud.com" \
DEFENSECLAW_OTEL_METRICS_PROTOCOL="http" \
DEFENSECLAW_OTEL_METRICS_URL_PATH="/v2/datapoint/otlp" \
DEFENSECLAW_OTEL_LOGS_ENDPOINT="ingest.us1.observability.splunkcloud.com" \
DEFENSECLAW_OTEL_LOGS_PROTOCOL="http" \
OTEL_SERVICE_NAME="pavan-defenceapp" \
./defenseclaw-gateway
```

Without explicit logs env vars, `resolveValue` falls back to the global `otel.endpoint`. If that is also empty (the default), the OTel SDK targets `localhost:4317` and silently drops data.

### Splunk HEC (splunk config block)

The audit store's HTTP Event Collector pipeline. Sends DefenseClaw audit events (scan results, block/allow decisions, alerts) to Splunk Enterprise or Splunk Cloud. Configured separately in `config.yaml`:

```yaml
splunk:
  enabled: true
  hec_endpoint: "https://splunk.example.com:8088/services/collector/event"
  hec_token: ""       # set via DEFENSECLAW_SPLUNK_HEC_TOKEN env var
  index: defenseclaw
  source: defenseclaw
  source_type: _json
```

HEC events are always in Splunk's native JSON format — they are not OTLP and do not flow through the OTel SDK.

## Metrics Exported

| Metric | Type | Description |
|--------|------|-------------|
| `defenseclaw.llm.tokens` | Counter | Token consumption (prompt + completion) |
| `defenseclaw.guardrail.evaluations` | Counter | Guardrail evals by action (block/allow/flag) |
| `defenseclaw.guardrail.latency` | Histogram | Evaluation latency (ms) |
| `defenseclaw.scan.*` | Counter | Scan operations |
| `defenseclaw.tool.*` | Counter | Tool invocations |
| `defenseclaw.alert.*` | Counter | Alert events |

## start.sh Options

| Flag | Description | Default |
|------|-------------|---------|
| `--enable-telemetry` | Enable direct-to-Splunk OTel telemetry | off |
| `--access-key TOKEN` | Splunk access token (required with `--enable-telemetry`) | — |
| `--realm REALM` | Splunk ingest realm | `us1` |
| `--app-name NAME` | OTel service name (appears in Splunk APM) | `defenseclaw` |

`start.sh` configures traces and metrics only. To also export OTel logs, add `DEFENSECLAW_OTEL_LOGS_ENDPOINT` and `DEFENSECLAW_OTEL_LOGS_PROTOCOL` via direct invocation (see [Logs & Events](#logs--events)).

## Environment Variables

### Global

| Env var | Effect | Default |
|---------|--------|---------|
| `DEFENSECLAW_OTEL_ENABLED` | Enable/disable OTel | `false` |
| `DEFENSECLAW_OTEL_ENDPOINT` | Global fallback endpoint | — |
| `DEFENSECLAW_OTEL_PROTOCOL` | Global fallback protocol (`grpc` or `http`) | `grpc` |
| `DEFENSECLAW_OTEL_TLS_INSECURE` | Skip TLS verification | `false` |
| `OTEL_SERVICE_NAME` | Override `service.name` resource attribute | `defenseclaw` |
| `SPLUNK_ACCESS_TOKEN` | Auto-injected as `X-SF-Token` header | — |

### Per-signal (override global)

| Env var | Effect |
|---------|--------|
| `DEFENSECLAW_OTEL_TRACES_ENDPOINT` | Traces endpoint |
| `DEFENSECLAW_OTEL_TRACES_PROTOCOL` | Traces protocol (`grpc` or `http`) |
| `DEFENSECLAW_OTEL_TRACES_URL_PATH` | Traces HTTP URL path override |
| `DEFENSECLAW_OTEL_METRICS_ENDPOINT` | Metrics endpoint |
| `DEFENSECLAW_OTEL_METRICS_PROTOCOL` | Metrics protocol (`grpc` or `http`) |
| `DEFENSECLAW_OTEL_METRICS_URL_PATH` | Metrics HTTP URL path |
| `DEFENSECLAW_OTEL_LOGS_ENDPOINT` | Logs endpoint |
| `DEFENSECLAW_OTEL_LOGS_PROTOCOL` | Logs protocol (`grpc` or `http`) |

Per-signal env vars take priority over global. If unset, the signal falls back to the global endpoint/protocol.

All env vars override `~/.defenseclaw/config.yaml` without modifying the file.

## Config File Reference (otel section)

`service.name` can also be set in config:

```yaml
otel:
  resource:
    attributes:
      service.name: pavan-defenceapp
```

Full otel section:

```yaml
otel:
  enabled: false
  protocol: grpc              # global default protocol
  endpoint: ""                # global default endpoint
  headers: {}
  tls:
    insecure: false
    ca_cert: ""
  traces:
    enabled: true
    sampler: always_on
    sampler_arg: "1.0"
    endpoint: ""              # per-signal override (empty = use global)
    protocol: ""              # per-signal override (empty = use global)
    url_path: ""              # HTTP URL path override
  metrics:
    enabled: true
    export_interval_s: 10
    endpoint: ""
    protocol: ""
    url_path: ""              # e.g., /v2/datapoint/otlp for Splunk
  logs:
    enabled: true
    emit_individual_findings: false
    endpoint: ""
    protocol: ""
  batch:
    max_export_batch_size: 512
    scheduled_delay_ms: 5000
    max_queue_size: 2048
  resource:
    attributes: {}
```
